# 受 Karpathy 启发的 Claude Code 指南（扩展版）

一个单一的 `SKILL.md` 文件，用于改善 Claude Code 的行为，源自 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876)，并扩展了更多规则。

[English](./README.md) | 简体中文

## 问题所在

来自 Andrej 的推文：

> "模型会代你做错误假设，然后不假思索地执行。它们不管理自身的困惑，不寻求澄清，不呈现矛盾，不展示权衡，在应该提出异议时也不反驳。"

> "它们真的很喜欢把代码和 API 搞复杂，堆砌抽象概念，不清理死代码……明明 100 行能搞定的事情，非要实现成 1000 行的臃肿架构。"

> "它们有时仍会改动或删除自己理解不足的代码和注释，即使这些内容与任务本身无关。"

## 解决方案

十二条原则，集中在一个文件中，直接解决这些问题及更多：

| 原则 | 解决什么问题 |
|------|-----------|
| **编码前思考** | 错误假设、隐藏困惑、缺少权衡 |
| **简洁优先** | 过度复杂、臃肿抽象 |
| **精准修改** | 无关编辑、触碰不应碰的代码 |
| **目标驱动执行** | 通过测试优先、可验证的成功标准 |
| **模型仅用于判断** | 将确定性逻辑交给 LLM |
| **Token 预算** | 静默超出上下文限制 |
| **呈现冲突** | 调和相互矛盾的模式 |
| **编码前先阅读** | 不理解调用方就添加代码 |
| **测试验证意图而非行为** | 测试不编码业务含义 |
| **检查点** | 任务中途失去状态感知 |
| **遵循代码库规范** | 静默分化代码风格 |
| **明确失败** | 压制错误和跳过的检查 |

## 十二条规则详解

### 规则 1 — 编码前思考

明确陈述假设。如果不确定，询问而非猜测。
存在歧义时呈现多种解释。
存在更简单方案时提出异议。
困惑时停下来，指出不清楚的地方。

### 规则 2 — 简洁优先

用最少的代码解决问题。不做推测性编码。
不添加要求之外的功能。不为一次性代码创建抽象。
检验标准：资深工程师会觉得这过于复杂吗？如果是，简化。

### 规则 3 — 精准修改

只碰必须碰的。只清理自己造成的混乱。
不要"改进"相邻的代码、注释或格式。
不要重构没坏的东西。匹配现有风格。

### 规则 4 — 目标驱动执行

定义成功标准。循环验证直到达成。
不要盲从步骤。定义目标并迭代。
强有力的成功标准让你能够独立循环执行。

### 规则 5 — 模型仅用于判断

模型适用于：分类、起草、总结、提取。
模型不适用于：路由、重试、确定性转换。
如果代码能回答，就用代码。

### 规则 6 — Token 预算不是建议

每任务：4,000 tokens。每会话：30,000 tokens。
接近预算时，总结并重新开始。
明确指出超支。不要静默超出。

### 规则 7 — 呈现冲突，不要调和

如果两种模式相互矛盾，选择一种（更新的 / 经过更多测试的）。
解释原因。标记另一种待清理。
不要混合矛盾的模式。

### 规则 8 — 编码前先阅读

添加代码前，先阅读导出、直接调用方、共享工具。
"看起来无关"是危险的。如果不确定代码为什么那样组织，先问。

### 规则 9 — 测试验证意图，而非仅验证行为

测试必须编码行为为什么重要，而不仅是什么行为。
如果业务逻辑变更时测试不会失败，那它就是错的。

### 规则 10 — 每个重要步骤后设置检查点

总结已完成项、已验证项、剩余项。
不要从你无法描述回来的状态继续。
如果迷路了，停下来重新梳理。

### 规则 11 — 遵循代码库规范，即使你不认同

代码库内，一致性 > 个人偏好。
如果你真的认为某个规范有害，提出来。不要静默分化。

### 规则 12 — 明确失败

如果有任何项被静默跳过，"已完成"就是错的。
如果有任何测试被跳过，"测试通过"就是错的。
默认呈现不确定性，而非隐藏它。

## 安装

**选项 A：Claude Code 插件（推荐）**

在 Claude Code 中，首先添加插件市场：
```
/plugin marketplace add onemott/andrej-karpathy-skills
```

然后安装插件：
```
/plugin install andrej-karpathy-skills@karpathy-skills
```

**选项 B：CLAUDE.md（按项目）**

新项目：
```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/onemott/andrej-karpathy-skills/main/CLAUDE.md
```

已有项目（追加）：
```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/onemott/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

## 在 Cursor 中使用

本仓库包含一个已提交的 Cursor 项目规则 ([`.cursor/rules/karpathy-guidelines.mdc`](.cursor/rules/karpathy-guidelines.mdc))，因此在 Cursor 中打开项目时同样适用这些指南。详情请参见 **[CURSOR.md](CURSOR.md)**，包括如何在其他项目中使用该规则，以及它与 Claude Code 的关系。

## 核心洞察

来自 Andrej：

> "LLM 非常擅长循环执行直到达成特定目标……不要告诉它该做什么，给它成功标准，然后看着它完成。"

"目标驱动执行"原则正是捕捉了这一点：将指令式指令转化为带有验证循环的声明式目标。

## 如何判断它在起作用

如果你看到以下情况，说明这些指南正在发挥作用：

- **diff 中不必要的改动更少** — 只有请求的改动出现
- **因过度复杂而导致的重写更少** — 代码第一次就写得简洁
- **澄清问题在实现之前提出** — 而不是在犯错之后
- **干净、精简的 PR** — 没有顺带的重构或"改进"
- **明确呈现权衡** — 冲突模式被标记，而非隐藏
- **预算意识** — Token 限制被遵守，而非静默超出

## 定制

这些指南设计用于与项目特定指令合并。将它们添加到你现有的 `CLAUDE.md` 或创建一个新的。

对于项目特定规则，添加如下章节：

```markdown
## 项目特定指南

- 使用 TypeScript 严格模式
- 所有 API 端点必须有测试
- 遵循 `src/utils/errors.ts` 中现有的错误处理模式
```

## 权衡说明

这些指南倾向于**谨慎而非速度**。对于琐碎的任务（简单的拼写错误修复、显而易见的一行修改），请自行判断 —— 并非每个改动都需要完整的严谨流程。

目标是减少非琐碎工作中的代价高昂的错误，而不是拖慢简单任务。
